# 조건 기반 유튜버 찾기

공개 정보만으로 후보를 넓게 찾은 뒤 실제 유튜브 페이지에서 최신 수치와 근거를 검증한다. 필수 조건은 점수로 상쇄하거나 사용자 허락 없이 완화하지 않는다.

## 필요한 리소스

- 검색어 확장이 필요하면 먼저 [references/search-strategies.md](references/search-strategies.md)를 읽는다.
- 조사 기록 또는 최종 응답을 만들기 전에 [references/output-schema.md](references/output-schema.md)를 읽는다.
- 수치 계산과 게이트 판정에는 `scripts/score_candidates.py`를 실행한다. 직접 암산하지 않는다.
- 계산된 JSON에서 최종 HTML 보고서를 만들 때는 `scripts/render_report.py`를 실행한다. HTML을 손으로 복제하지 않는다.
- 유튜브 페이지를 열기 전에 설치된 `control-in-app-browser` 스킬의 `SKILL.md`를 끝까지 읽고 그 지침을 따른다. 공식 인앱 Browser를 시작했다는 사실을 사용자에게 알린다.
- Browser를 시작하기 직전에 선택한 브라우저의 `documentation()`에 정의된 탭 목록 API로 기존 탭 식별자를 기록한다. 조사와 보고서 검수가 끝나면 이번 작업에서 새로 연 탭만 닫고 같은 API로 닫힘을 확인한다. 기존 사용자 탭이나 로컬 브라우저 전체를 닫지 않는다. 전용 브라우저를 시작했다면 Browser 스킬 지침에 따라 종료 여부를 확인하고 정리한다.

## UI 진행 상태 연동

프롬프트에 `UI_RUN_ID`, `UI_STATUS_PATH`, `UI_SCORED_PATH`, `UI_REPORT_PATH`가 있으면 [references/output-schema.md](references/output-schema.md)의 UI 상태 규약을 따른다.

1. 네 경로가 현재 작업공간의 `ui/public/research-runs/<UI_RUN_ID>/` 아래인지 확인한다. 바깥 경로나 `..`가 있으면 쓰지 않는다.
2. 조건을 확정하면 `criteria`, 후보 수집을 시작하면 `discovery`, YouTube 직접 검증을 시작하면 `verification`, 스코어러를 실행하면 `scoring`, HTML 생성을 시작하면 `reporting` 상태를 기록한다.
3. 상태가 바뀌거나 실제 후보·검증 수가 늘어난 주요 시점에만 `status.json`을 갱신한다. 로그에는 실제 완료한 작업만 적고 예정 작업을 완료처럼 쓰지 않는다.
4. 중단되면 가능한 경우 `error` 상태와 실제 원인을 기록한다. CAPTCHA, 로그인, 사용자 확인이 필요하면 우회하지 않는다.
5. 최종 계산 JSON과 HTML을 지정된 UI 경로에도 복사한 뒤 파일 존재를 확인하고 마지막으로 `complete` 상태를 기록한다.

## 1. 조건을 구조화한다

자연어 요청을 다음 필드로 정리한다.

`product`, `category`, `market`, `language`, `target_audience`, `subscriber_min`, `subscriber_max`, `recent_video_count`, `recent_content_type`, `comment_rule`, `minimum_comment_total`, `minimum_average_comments`, `minimum_average_views`, `upload_recency_days`, `sponsored_content_required`, `preferred_content`, `excluded_content`, `result_count`, `strictness`.

지정되지 않은 값에는 다음 기본값을 적용한다.

- `market`: 한국
- `language`: 한국어
- `recent_video_count`: 3
- `recent_content_type`: `longform`(채널의 동영상 탭에 있는 최신 일반 영상)
- `subscriber_max`: 100000
- `result_count`: 10
- `upload_recency_days`: 90
- `sponsored_content_required`: false. 단, 검증된 경험자를 순위에서 우선한다.
- `strictness`: `strict`

상품이나 필수 게이트의 의미를 안전하게 정할 수 없을 때만 질문한다. 질문은 한 번에 꼭 필요한 것만 최대 3개로 제한한다. 기본값으로 해결되는 내용은 묻지 않는다. “댓글 500개”가 합계인지 영상별·평균인지 문맥상 불명확하면 확인한다. 문맥이 최근 3개 댓글 기준이면 기본적으로 합계로 해석하되 이 해석을 조사 기준에 명시한다.

일반 영상과 Shorts를 같은 표본에 섞지 않는다. 사용자가 전체 업로드를 요청해도 `longform`과 `shorts`를 별도 코호트로 계산하고 결과를 분리한다. 사용자가 “최근 영상”이라고만 하면 `longform`으로 해석한다.

조사를 시작하기 전에 기준을 한 문장으로 요약하고 진행 상황을 알린다.

## 2. 상품과 타깃을 해석한다

상품명 일치 영상에만 매달리지 말고 구매 상황, 사용 맥락, 타깃의 일상, 인접 카테고리, 경쟁 제품으로 확장한다. 시청자 성별·연령은 공개 자료 없이 단정하지 않는다. 콘텐츠에 근거한 판단은 반드시 “추정”이라고 표시하고 근거를 적는다.

브랜드 안전성에서 논란, 위험 표현, 경쟁 브랜드 독점 가능성을 공개 근거로 점검한다. 확인되지 않은 광고 단가, 연락처, 매출 효과는 만들지 않는다.

## 3. 초기 후보를 넓게 수집한다

먼저 공개 웹 검색을 사용한다. 검색 결과는 후보 발견 용도로만 쓰며 현재 구독자·댓글·조회수의 최종 근거로 쓰지 않는다.

상품, 리뷰, 추천, 사용 상황, 브이로그, 타깃, 유료광고, 협찬, 신제품, 인접 카테고리, 경쟁 제품 조합을 여러 개 사용한다. 최종 목표 수의 최소 3배를 1차 후보로 수집한다. 후보가 중복되면 채널 URL 기준으로 합친다. 초기 후보 수집이 끝나면 후보 수와 주요 검색 축을 짧게 알린다.

## 4. Browser Use로 직접 검증한다

Playwright MCP나 임의의 외부 브라우저 자동화로 대체하지 않는다. 공식 Browser Use를 사용하고 다음 원칙을 지킨다.

1. 실제 채널 페이지를 열어 정확한 채널명, 채널 URL, 현재 공개 구독자 수를 기록한다.
2. 요청한 콘텐츠 탭에서 최신순으로 영상을 확인한다. 고정 영상, 라이브 다시보기, 멤버십 전용, 운영 공지 등 부적합 항목은 제외하고 이유를 기록한다.
3. 각 유효 영상 페이지를 열어 제목, 실제 URL, 게시일, 조회수, 공개 댓글 수를 확인한다.
4. 영상 설명에서 `유료광고`, `부분 유료광고`, `협찬`, `제품 제공` 등의 실제 근거를 확인한다. 단순 언급과 광고 경험을 구분한다.
5. 관련 제품 콘텐츠, 업로드 주기, 콘텐츠 분위기, 예상 시청자, 상품을 자연스럽게 넣을 수 있는 아이디어를 기록한다.
6. 최종 후보 URL은 실제로 열어 확인한 주소만 사용한다.

첫 후보의 검증 결과가 나오면 확인 가능한 데이터와 막힌 항목을 짧게 공유한다. CAPTCHA·사람 확인은 우회하지 않는다. 로그인, 비밀번호, MFA 또는 계정 접근이 필요하면 중단하고 공개 정보로 확인할 수 없다고 표시한다. 같은 페이지를 무한 반복하지 않는다.

댓글 비활성화나 비공개 수치는 추정하지 말고 `null`과 “확인 불가”로 기록한다. `1.2천`처럼 축약된 값은 실제 영상 페이지에서 더 정확한 공개 값을 찾고, 정확한 값이 끝내 없으면 표시 문자열과 근사치라는 사실을 별도 기록한다. 근사치를 엄격한 경계값의 PASS 근거로 쓰지 않는다.

## 5. 계산하고 필수 조건을 판정한다

후보 데이터를 최신 영상부터 배열한 JSON으로 저장하고 다음처럼 실행한다.

```text
python scripts/score_candidates.py --input candidates.json --pretty
```

스크립트에는 `evaluation_date`를 확인 날짜로 전달한다. `recent_content_type`에는 `longform` 또는 `shorts`만 전달한다. 전체 업로드는 두 번 나누어 계산한다.

계산식은 다음과 같다.

- `comment_total = 최근 N개 유효 영상 댓글 수의 합`
- `average_comments = comment_total / N`
- `average_views = 최근 N개 유효 영상 조회수 합 / N`
- `engagement_by_views = comment_total / 최근 N개 조회수 합 * 100`

하나라도 필요한 값이 없거나 N개 표본이 완성되지 않으면 해당 집계값을 임의의 0이나 부분 합계로 만들지 않는다.

판정은 다음과 같다.

- `PASS`: 모든 필수 게이트와 증거 요건을 충족한다.
- `NEAR MATCH`: 검증이 완료되었고 필수 수치 조건 하나만 근소하게 미달한다. PASS로 승격하지 않는다.
- `FAIL`: 둘 이상의 필수 조건이 미달하거나 하나가 크게 미달한다.
- `UNVERIFIED`: 필수 공개 정보나 근거가 부족해 판정할 수 없다.

필수 게이트는 추천 점수로 상쇄하지 않는다. `strictness: flexible`도 자동 완화를 뜻하지 않으며 근접 후보 설명 범위에만 영향을 준다.

## 6. 광고 적합성을 평가한다

공개 근거가 있는 경우에만 각 항목을 0~5점으로 평가한다. 모르면 `null`로 둔다.

- Product Fit: 30%
- Audience Fit: 20%
- Engagement: 20%
- Reach: 15%
- Advertising Experience: 10%
- Activity: 5%

Commercial Fit, Brand Safety, Authenticity도 별도로 0~5점 또는 `null`로 기록하고 상세 분석에 반영한다. 가중 점수는 여섯 기본 항목이 모두 확인된 경우에만 계산한다. 상품 적합성이나 시청자 적합성 추정에는 근거를 붙인다.

조건이 강해 PASS가 부족할 것으로 보이면 그 사실을 중간에 알리되 조건을 바꾸지 않는다.

## 7. 교차 검증하고 출력한다

최종 검증을 시작한다고 알린 뒤 각 PASS 후보를 다시 열어 다음 증거를 재확인한다.

- 채널 페이지와 현재 구독자 수
- 요청한 유형의 최근 N개 영상 링크
- 영상별 공개 댓글 수, 댓글 합계, 평균 조회수
- 상품 관련성 근거
- 광고·제품 소개 경험 근거 또는 “확인되지 않음”
- 확인 날짜

증거가 부족하면 PASS에서 `UNVERIFIED`로 내린다. 순위는 PASS 안에서만 가중 점수, 핵심 적합성, 근거 완전성을 함께 사용한다. 점수가 `null`이면 억지로 수치 순위를 만들지 말고 정성 근거를 명시한다.

최종 응답은 조사 기준 한 문장, PASS 표, 후보별 상세 분석, NEAR MATCH 표, FAIL/UNVERIFIED 요약, 최종 집계와 다음 행동 순서로 작성한다. 채널명과 영상명에 검증된 유튜브 URL을 연결한다. 확인 날짜를 표시한다.

PASS가 없으면 “현재 검증한 후보 중 모든 조건을 충족한 채널은 찾지 못했다”고 명확히 말하고 조사한 수, 가장 잦은 탈락 조건, NEAR MATCH 최대 5명, 가능한 완화와 예상 영향을 제안한다. 사용자 허락 전에는 완화 후보를 PASS로 표시하지 않는다. 일반 영상 결과가 크게 달라질 가능성이 있을 때만 별도의 Shorts 후보군 조사를 다음 행동으로 제안한다.

## 8. HTML 보고서를 생성하고 검수한다

조사가 끝나면 Markdown 응답과 별도로 자체 포함형 HTML 보고서를 기본 생성한다. 사용자가 명시적으로 파일 생성을 원하지 않는다고 했을 때만 생략한다.

1. `score_candidates.py` 출력 JSON에 `report_metadata`와 후보별 정성 분석 필드를 채운다. 수치·상태를 손으로 다시 계산하거나 변경하지 않는다.
2. 다음 명령으로 HTML을 만든다.

```text
python scripts/render_report.py --input scored-candidates.json --output influencer-report.html
```

3. 보고서에는 조사 기준, 확인일, 조사 후보 수, PASS/NEAR MATCH/FAIL/UNVERIFIED 집계, 후보 표, PASS와 근접 후보의 상세 근거, 최근 영상 링크와 수치, 주요 탈락 원인, 다음 행동, 방법론·한계를 포함한다.
4. 채널·영상 링크는 실제로 열어 확인한 HTTPS 유튜브 주소만 활성화한다. 렌더러는 다른 도메인 URL을 링크로 만들지 않는다.
5. 외부 CSS·JavaScript·폰트·이미지 없이 한 파일로 생성한다. 모든 동적 문자열은 HTML 이스케이프하고 모바일·인쇄 레이아웃을 유지한다.
6. Browser Use로 생성한 로컬 HTML을 열어 제목, 표, 상태 집계, 링크, 긴 채널명, 모바일 폭을 시각적으로 검수한다. 깨진 링크나 잘린 콘텐츠가 있으면 수정하고 다시 생성한다.
7. 검수가 끝나면 조사 시작 전에 기록한 탭과 비교하여 이번 작업에서 연 유튜브·검색·로컬 보고서 탭만 닫는다. 잠시 기다린 뒤 선택한 브라우저 문서의 탭 목록 API로 남은 작업 탭이 없는지 확인한다.
8. 최종 응답에는 HTML 파일의 클릭 가능한 경로를 제공하고, 보고서가 어느 날짜의 공개 정보에 근거하는지 밝힌다.
