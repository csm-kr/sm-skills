# 웹 리서치 (Phase 2)

인테이크에서 **근거가 빈 주장**, **최신 데이터가 필요한 지점**, 또는 사용자의 "웹에서 찾아서 PT 만들어줘" 요청을 웹으로 보강한다. 웹 리서치는 슬라이드 빌드 전 단계이며, 결과는 반드시 소스 맵으로 남긴다.

## 트리거
- 사용자가 "웹에서 찾아서", "리서치해서", "최신/요즘 자료로", "근거 찾아서"라고 말한다.
- 인테이크 노트의 근거가 `[리서치 필요]`로 남아 있다.
- 수치, 순위, 가격, 일정, 정책, 최근 사례, 시장/트렌드 주장이 들어간다.
- 특정 회사/인물/제품/기술의 현재 상태를 말해야 한다.

## 리서치 절차
1. **질문을 리서치 쿼리로 쪼갠다.** 발표 목적과 청중 기준으로 `must-have claims` 3~7개를 만든다.
2. **웹으로 확인한다.** 최신성이나 사실 정확도가 필요한 claim은 반드시 검색/브라우징으로 확인한다.
3. **소스 맵을 만든다.** 각 항목을 `claim / source URL / source type / date checked / slide-use`로 기록한다.
4. **슬라이드화한다.** 한 슬라이드에는 한 claim만 넣고, 근거가 약한 claim은 표현을 낮추거나 제외한다.
5. **출처를 남긴다.** 슬라이드에는 짧게, 작업 노트나 `assets/CREDITS.txt`에는 URL과 접근일을 남긴다.

## 소스 품질
- 우선순위: 공식/원문/데이터셋/논문/공시/기관 보고서 > 신뢰 가능한 언론/전문 매체 > 블로그/마케팅 페이지.
- 회사·제품 기능은 공식 문서나 공식 블로그를 우선한다.
- 통계는 조사기관, 원자료, 방법론이 보이는 자료를 우선한다.
- 광고성 페이지, 출처 없는 인용, 숫자만 떠도는 글은 핵심 근거로 쓰지 않는다.
- 서로 충돌하는 자료가 있으면 최신 날짜와 원자료 여부를 비교하고, 충돌 사실을 노트에 남긴다.

## 웹에서 찾아서 PT 만들기 모드
사용자가 자료를 주지 않고 웹 리서치 기반 제작을 요청하면 바로 HTML을 만들지 않는다. 먼저 아래 짧은 형태로 리서치 방향을 확정한다.

```text
목적:
청중:
핵심 질문:
필수로 확인할 claim:
쓰지 말아야 할 자료/톤:
```

사용자가 빠르게 진행하라고 하면 합리적으로 가정하고 진행하되, 소스 맵에 `assumption`을 표시한다. 리서치 후에는 슬라이드 개요를 먼저 제안하고, 합의 뒤 빌드한다.

## 슬라이드 반영 규칙
- 숫자·날짜·비교·순위는 출처 없는 상태로 슬라이드에 넣지 않는다.
- 최신성 있는 문장에는 기준일을 붙인다. 예: `2026-07-09 확인 기준`.
- 근거가 웹 페이지 하나뿐이면 강한 일반화 문장을 피한다.
- 출처가 많아도 슬라이드에는 핵심 사실만 남긴다. 본문 덤프 금지.
- 이미지가 필요하면 먼저 공식/공개/라이선스 가능한 이미지를 찾고, 없으면 설명형 SVG/도식 또는 승인받은 생성 이미지를 쓴다.
- 실제 지도가 필요하면 직접 그리지 말고 공식/공개/라이선스 가능한 지도 베이스를 웹에서 찾는다. 국가·도시·여행 루트·언어권·음식권·역사 지점은 지도 베이스 위에 핀, 루트, 반투명 영역, 라벨을 overlay로 그린다. 출처와 라이선스는 `assets/CREDITS.txt`에 기록한다.

## 소스가 많을 때
- 독립적인 소스가 3개 이상이면 주제별로 나누어 요약한다.
- 각 요약은 `URL + 핵심 사실 3줄 + 슬라이드 후보 1개`만 남긴다.
- 긴 기사나 보고서를 그대로 붙여 넣지 않는다.

## 실패 처리
- 근거를 못 찾으면 "근거 없음"으로 표시하고, 슬라이드에서 빼거나 사용자에게 확인한다.
- 최신 정보가 검색 결과마다 다르면 단일 결론으로 단정하지 않는다.
- 리서치가 길어질 때는 먼저 1차 소스 맵과 개요를 보여주고 계속 진행 여부를 묻는다.
