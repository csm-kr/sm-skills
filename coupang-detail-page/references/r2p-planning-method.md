# R2P 근거·질문·페이지 설계법

## 목적

R2P는 `RAW → REFERENCE → REASON → PROOF → PAGE`를 한 줄로 연결하는 상세페이지 기획법이다. 고정된 10장 역할을 채우는 대신, 현재 판매 상품에서 확인 가능한 사실과 구매자가 실제로 결정해야 할 질문만 남긴다. 디자인은 사실을 이해시키는 표현 수단이며 상품 사실의 출처가 아니다.

## 1. RAW — 현재 상품의 정체성을 잠근다

모든 입력 자산에 ID와 역할을 먼저 부여한다.

| 역할 | ID 예 | 의미 | 허용 범위 |
|---|---|---|---|
| `RAW_PRIMARY` / `RAW_DETAIL` | `A01` | 사용자가 제공한 현재 판매 상품의 주 실사·디테일 | 외형·색·부품·라벨·관찰 가능한 구조 |
| `RAW_MEASUREMENT` | `A02` | 현재 상품과 자가 함께 보이는 실측 사진 | 사진에서 판독 가능한 치수만 |
| `RAW_DEMO` | `A03` | 동일 상품·조건·판정이 이어지는 실사 동작·시험 | 촬영 조건에서 직접 보인 구조·동작·결과만 |
| `WEB_MATCH` | `A04` | M1 또는 M2로 현재 제품과 연결한 웹 보조 자산 | M1+E1/E2는 명시 사실 후보, M2+E3는 원본과 충돌 없는 외형 보조만 |
| `REF_STRUCTURE` / `REF_MOOD` / `REF_TYPO` / `REF_PHOTO` | `A05` | 실제 상세페이지·에디토리얼의 표현 참고 | 추상적 구조·색·위계·사진 리듬만 |
| `GENERATED` | `A06` | 승인 뒤 만든 정적·동적 출력 | 결과물로만 사용하며 상품 사실 근거로 사용 금지 |

- `RAW_*`와 조건을 충족한 `WEB_MATCH`는 상품 계열이고 `REF_*`는 표현 계열이다. 둘을 서로 대체하지 않는다.
- AI 생성 결과, 이전 페이지 출력, 유사 상품 이미지는 `RAW_*`가 될 수 없다.
- 사진에 두 개가 보인다는 이유만으로 판매 구성을 두 개·한 쌍으로 확정하지 않는다.
- 자가 없는 사진 비율, 유사 상품 표, 리뷰 문장으로 치수를 추정하지 않는다.
- 원본과 공식 정보가 충돌하면 충돌을 기록하고 승인 전까지 해당 주장을 보류한다.

## 2. REFERENCE — 베끼지 않고 표현 원리만 추출한다

각 `REF_*` 자산에서 아래 다섯 항목만 추출한다.

```text
REF_PRINCIPLE_ID: RP01
REFERENCE_ASSET_ID: A05
출처·접근일:
채택 원리: 타이포 위계 / 여백 / 카드 밀도 / 사진 리듬 / 색 대비 중 구체 항목
적용 장:
복제 금지: 실제 문구 / 수치 / 로고 / 제품 기능 / 고유 그래픽 / 동일 구도
```

레퍼런스가 없으면 페이지 계약의 `REF_PRINCIPLE_IDS`를 `NONE`으로 기록하고 자체 설계 이유를 남긴다. 억지로 유사 상품을 레퍼런스로 지정하지 않는다. 한 페이지가 여러 레퍼런스를 혼합할 때도 원리별 출처를 분리한다.

## 3. REASON — 장점이 아니라 구매 질문을 세운다

먼저 아래 순서로 질문 후보를 만든다.

1. `기능`: 어떤 불편을 어떤 확인된 구조·작동으로 줄이는가
2. `적합`: 내 몸·공간·기기·용도에 맞는가
3. `실측·사양`: 실제 크기·용량·무게·재질은 무엇인가
4. `구성`: 주문하면 정확히 무엇이 몇 개 오는가
5. `사용`: 설치·착용·세척·관리·보관은 어떻게 하는가
6. `안전·한계`: 호환 불가·금지 조건·주의사항은 무엇인가
7. `선호`: 색·스타일·분위기 중 무엇을 고를 수 있는가

각 질문을 `INFO_ID` 하나로 만들고, 그 질문에만 속하는 `Fact ID`를 배정한다. 같은 사실을 문제·비교·추천·활용·CTA에서 바꿔 말한 것은 한 질문이다. 구매자가 답을 얻기 위해 별도의 사진·도식·행동이 필요할 때만 새 질문으로 분리한다.

### 의사결정 단계

| 단계 | 질문 | 주로 쓰는 정보 |
|---|---|---|
| `NOTICE` | 나도 이 불편을 겪는가 | 검증된 대표 불편과 문제 장면 |
| `UNDERSTAND` | 이 상품은 어떤 방식으로 답하는가 | 핵심 기능·구조·상품 소개 |
| `VERIFY` | 그 말을 무엇으로 확인할 수 있는가 | 원본 디테일·공식 사양·실사 시험 |
| `FIT` | 내 조건에 맞는가 | 실측·호환·옵션·구성 |
| `USE` | 실제로 어떻게 쓰고 관리하는가 | 순서·주의·세척·보관 |
| `DECIDE` | 남은 위험 없이 선택 가능한가 | 마지막 고유 정보와 중립 CTA |

기획은 `NOTICE → UNDERSTAND → VERIFY/FIT/USE → DECIDE` 흐름을 따르되, 상품에 없는 단계를 억지로 만들지 않는다.

## 4. PROOF — 카피보다 먼저 보일 증거를 정한다

각 정보 단위마다 `PROOF_ID`와 최소 충분 증거를 하나 지정한다.

| 증거 형태 | 적합한 질문 | 금지 확대 |
|---|---|---|
| 원본 디테일 컷 | 봉제·버튼·입구·표면·구성 | 사진에 안 보이는 소재 성능 |
| 실측 사진·도식 | 길이·폭·높이·호환 크기 | 자 없는 추정값·유사 모델 치수 |
| 공식 사양 표 | 재질·용량·전력·호환·주의 | 다른 SKU·옵션의 사양 전이 |
| 사용 순서 | 착용·설치·열기·세척 | 사용법을 성능 입증으로 표현 |
| 실사 비교 시험 | 고정력·누수·작동처럼 판정 가능한 결과 | 미기록 조건의 보편적 보장 |
| AI 비교 연출 | 사용자가 확인한 정성적 불편과 구조의 이해 | 실제 비교 시험·수치·객관적 우위 |
| 생활 장면 | 사용 맥락·크기 감각·스타일 | 효능·성능의 증거로 사용 |

`COPY_CLAIM`의 강도는 `PROOF`보다 강할 수 없다. 냉감·차단·방수·내구·항균·보호처럼 성능을 뜻하는 단어는 사용자 확인, 동일 모델 공식 근거 또는 실제 시험 범위가 없으면 보류한다.

실제 비교 사진이 없어도 사용자 확인을 받은 정성적 불편과 현재 상품 구조는 정적 `AI_ILLUSTRATION`으로 설명하는 것을 기본값으로 둔다. 이때 비교 대상은 무브랜드 일반형으로 만들고, `PROOF_MODE: AI_ILLUSTRATION`과 `실제 시험 근거가 아님`이라는 경계를 기록한다. AI 연출을 실제 비교 시험, 객관적 우위, 수치나 신체 변화의 증거로 사용하지 않는다. 실증 비교를 원할 때만 동일 조건의 실제 촬영을 추가로 요청한다.

## 5. PAGE — 정보 단위 하나에 페이지 계약 하나를 준다

각 페이지는 아래 계약을 가져야 한다.

```text
PAGE_NO / ROLE_ID / DECISION_STAGE / INFO_ID
BUYER_QUESTION / PRIMARY_FACT / PROOF_ID
RAW_ASSET_IDS / REF_PRINCIPLE_IDS / REFERENCE_PRINCIPLE
SHOT_ID / SCENE_ID / LAYOUT_ID
EXACT_COPY / CLAIM_BOUNDARY / MOTION_ROLE
```

- `정보 단위 수 = 선정 장수`이며 기본 범위는 3~10장, 보통 5~8장이다. 근거가 세 개뿐이면 세 장으로 끝낸다.
- 첫 장은 `PROBLEM_HOOK`, 상품 소개는 정확히 한 장, 마지막은 새 정보와 CTA를 결합한다.
- 기능이 있으면 첫 장과 02 또는 03에서 그 기능이 답하는 불편과 작동 이유를 우선한다.
- 한 장을 가렸을 때 사라지는 구매 답이 없으면 삭제하거나 합친다.
- 동일 `PRIMARY_FACT`, 구매 질문, 제품 크롭, 장면, 지배 레이아웃을 재사용하지 않는다.
- 모든 문자열을 먼저 확정하고 이미지 생성에는 확정 문자열만 전달한다.

## 자산-페이지 계보 표

`outputs/<project-no>/asset-map.md`에 아래 네 표를 만든다.

```text
Asset Registry:
ASSET_ID / ROLE / ORIGIN / PATH_OR_URL / PRODUCT_MATCH / SOURCE_GRADE / SHA256 / OBSERVABLE_FACTS / ALLOWED_USE / FORBIDDEN_TRANSFER / STATUS

Evidence Ledger:
PROOF_ID / FACT_IDS / CLAIM_CLASS / EVIDENCE_TYPE / SOURCE_ASSET_IDS / STRENGTH / SKU_OPTION_SCOPE / ALLOWED_COPY / FORBIDDEN_EXPANSION / STATUS

Reference Principle Ledger:
REF_PRINCIPLE_ID / REFERENCE_ASSET_ID / REFERENCE_TYPE / ABSTRACT_PRINCIPLE / TARGET_PAGE_IDS / DO_NOT_COPY / STATUS / REJECTION_REASON

Page Source Contract:
장 / INFO_ID / DECISION_STAGE / RAW_ASSET_IDS / REF_PRINCIPLE_IDS / PROOF_ID / PRIMARY_FACT_IDS / PROOF_MODE / EXCLUSIVE_GAIN / DROP_TEST / CLAIM_BOUNDARY
```

웹에서 내려받아 로컬에 저장한 `WEB_MATCH`와 `REF_*`는 `web-research.md` 검색 기록 표의 마지막 `ASSET_ID` 열에도 같은 ID를 적는다. 한 행에는 정확히 한 `ASSET_ID`만 적고, 해당 행에는 `검색 결과 없음`이 아니라 canonical 직접 `http(s)` URL이 있어야 한다. 문서 상단의 `조사일`은 유효한 `YYYY-MM-DD`여야 한다. 서로 다른 로컬 SHA256 자산을 같은 direct image URL에 연결하지 않는다. 같은 HTML 상품 페이지에서 여러 컷을 얻은 경우에는 자산별 행을 만들고 실제 이미지 URL 또는 파생·캡처 관계를 각각 기록한다. 로컬 웹 자산이 이 역링크를 갖지 않으면 `validate_asset_map.py`가 실패한다.

검증 규칙:

- 모든 선정 장은 Page Source Contract에 정확히 한 번 등장한다.
- 로컬 자산 폴더를 역할과 일치시킨다: `RAW_PRIMARY`·`RAW_DETAIL`은 `original-images/`, `RAW_MEASUREMENT`·`RAW_DEMO`는 `evidence/`, `WEB_MATCH`는 `web-confirmed/`, `REF_*`는 `real-references/`에 둔다.
- `RAW_ASSET_IDS`는 Asset Registry의 상품 정체성 자산 또는 `NO_PRODUCT`여야 한다. `NO_PRODUCT`는 제품을 화면에 보여주지 않는 01 문제 장면에만 허용한다.
- `REF_PRINCIPLE_IDS`는 실제 RP ID 또는 `NONE`이어야 한다.
- `PROOF_ID`와 Fact ID는 실제 원장에 존재해야 한다.
- `REF_*`를 `RAW_ASSET_IDS`나 증거로 쓰면 실패한다.
- 같은 `PROOF_ID`를 여러 장에서 재사용하면 자동 실패한다.

## 카테고리별 첫 질문

| 카테고리 | 디자인보다 먼저 확인할 것 | 대표 증거 |
|---|---|---|
| 의류·착용품 | 조임·핏·실측·착용 방향·세탁 | 착용 전후 상황, 평면 실측, 봉제 디테일 |
| 용기·주방 | 용량·개폐·누수 조건·세척·구성 | 작동 순서, 치수 도식, 분해 컷 |
| 전자·충전 | 호환 규격·입출력·설치·발열/안전 주의 | 포트·라벨, 공식 사양, 설치 데모 |
| 거치·기계 | 장착 범위·조절 축·하중 조건·안전 | 실측, 설치 순서, 조건이 기록된 실사 시험 |
| 화장품·위생 | 성분·용량·사용 부위·방법·주의 | 라벨, 전성분/공식 설명, 사용 순서 |
| 식품·영양 | 내용량·단위 구성·영양·알레르기·보관 | 포장 라벨, 영양표, 구성 플랫레이 |
| 생활·청소 | 표면 적합성·사용법·세척/교체·보관 | 실제 작업 순서, 재질 디테일, 관리 컷 |
| 반려동물·유아 | 대상 크기·섭취/사용량·안전·보관 | 공식 급여/사용 기준, 라벨, 크기 비교 |

## 모션 결정

정지 화면보다 `순서`, `상태 변화`, `판정 가능한 결과` 중 하나를 더 잘 보여줄 때만 모션으로 승격한다.

- `REAL_DEMO`: 착용·설치·접기·열기·세척처럼 화면에서 보이는 동작
- `REAL_TEST`: 동일 상품, 기록된 조건, 연속 원본, 판정 기준이 있는 물리 시험
- `AI_ILLUSTRATION`: 구조 이해용 연출이며 성능 증거가 아님
- `NO_PROOF_HOOK`: 근거가 없어서 증명 영상을 만들지 않는 정직한 상태

모션의 가장 좋은 위치는 “주장 직후, 세부 사양 직전”이다. 문제 장면만 과장하거나 정지 이미지를 확대하는 장식 모션은 만들지 않는다.

## 완료 판정

다음을 모두 만족해야 `plan-review.html`을 사용자에게 보여줄 수 있다.

1. 동일·유사 상품 및 접근 가능한 상세페이지 3개 이상을 조사했다.
2. Fact 원장과 자산 계보 표가 완성됐다.
3. 기능·실측·구성·사용·주의 중 상품에 해당하는 질문이 디자인보다 먼저 배치됐다.
4. 페이지마다 서로 다른 구매 질문·Fact·Proof·컷·레이아웃이 있다.
5. 반복 전용 비교·추천·요약·CTA를 삭제했다.
6. 기획·자산 매핑·모션 검증을 통과했다.
7. HTML 승인 전에는 정적 이미지·GIF·영상을 생성하지 않았다.
