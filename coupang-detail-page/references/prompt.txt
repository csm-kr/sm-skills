# 쿠팡 상세페이지 기획 기준

이 문서는 사용자가 제공한 `prompt.txt`의 핵심을 스킬 배포용으로 정리한 강제 계약이다. 10장을 채우기 위한 반복보다 구매 판단에 필요한 서로 다른 정보를 우선한다.

## 1. 목표와 기본 규격

- 제품 정보에 근거한 쿠팡 모바일 상세페이지를 만든다.
- 기본 규격은 장마다 `800x2400` 세로형 PNG다.
- 기본 범위는 3~10장, 보통 5~8장이지만 목표 장수는 없다. `중복 제거 뒤 독점 정보 단위 수 = 선정 장수`로 정한다.
- 각 장은 독립적으로 읽히고, 전체를 이어 보면 자연스러운 구매 설득 흐름을 이룬다.
- 한 장은 큰 포스터 한 줄이 아니라 3~6개 세로 정보 구역으로 설계하되, 구역마다 한 메시지만 전달한다.
- 전체 장에서 같은 브랜드·폰트·팔레트·카드·사진 문법을 유지한다.
- 검증되거나 사용자가 명시적으로 확인한 기능·구조·실측·구성·사용법을 디자인·색·무드보다 먼저 보여준다.
- 새 정보 없이 기존 장점을 바꿔 말하는 비교·추천·활용·요약·독립 CTA 장은 삭제하거나 앞 장에 병합한다.

## 2. 사실 입력과 실측 게이트

기획 전에 다음 정보를 확인한다.

```text
상품명 / 카테고리 / 브랜드 / 모델명·SKU / 라벨 표기
핵심 장점 / 타깃 고객 / 대표 불편 / 사용 상황 / 사용 방법
구성품 / 판매 구성 수량 / 소재 / 색상 / 중량 / 제조국
실측 전체 길이 / 상단 입구 펼친 폭 / 몸판 펼친 폭 / 커프 길이
실측 사진 또는 정확한 모델의 공식 치수표
관리 방법 / 주의사항 / 검증된 차별점 / 강조할 분위기
객관적 수치·인증·시험 정보 또는 없음
실물 샘플·실사 촬영 가능 여부 / 실제 시험 원본·촬영 조건 또는 없음
금지 표현
```

- 입력하지 않은 사실은 문구나 시각적 암시로 만들지 않는다.
- 실측값은 현재 판매 제품을 늘리지 않고 평평하게 놓아 사용자가 잰 값과 자 사진, 또는 모델·옵션이 정확히 일치하는 `M1+E1/E2` 공식 표만 허용한다. 자 사진 없는 사용자 값은 `USER_DECLARED_SPEC / USER_CONFIRMED_NO_PHOTO`로 표시하고 자 사진·공식 도면처럼 연출하지 않는다.
- 유사 제품 치수, 사진 비율, 후기, 생성 이미지로 크기·착용 가능 팔둘레·프리사이즈를 추정하지 않는다.
- 웹 썸네일의 `1+1`, `2SET` 같은 표기와 원본에 보이는 물건 수가 판매 구성과 다를 수 있으면 사용자 확인 전 수량 카피를 금지한다.

## 3. 기능 우선 게이트

```text
핵심 기능 소구: 가장 중요한 기능 하나 또는 없음
기능 근거 Fact ID:
기능이 답하는 구매 불편:
디자인 보조 소구:
기능 우선 적용 장:
기능 없음 사유: 기능이 없을 때만
기능 우선 상태: 완료
```

- 기능에는 성능뿐 아니라 구조, 사용 편의, 안전, 구성과 시간·노력 절감 같은 실용적 구매 이유를 포함한다.
- 기능이 확인되면 `고객 불편 → 기능적 답 → 구조·사용 증거 → 실측·구성 → 디자인 보조` 순서로 설득한다.
- 사용자가 정성적 기능명만 확인한 경우 그 이름은 쓸 수 있지만 수치·지속 시간·작동 원리·우위·보장으로 확대하지 않는다.
- 기능 근거가 없으면 만들지 않고 관찰 가능한 구조·사용법·구성·실측을 먼저 다룬다.

## 4. 필수 웹 조사

- 동일 제품과 유사 제품을 각각 검색한다.
- 접근 가능한 실제 상세페이지를 최소 3개 비교해 첫 훅, 정보 순서, 실측·사양·관리, 디테일 캡션, 타이포 계층과 혼잡 요소를 기록한다.
- 동일 제품은 모델·라벨·패키지·외형을 확인하고 `M0~M3` 동일성, `E1~E4` 출처와 직접 URL을 남긴다.
- 유사 제품은 고객 질문, 정보 순서, 촬영 장면과 카테고리 표현 연구에만 사용한다.
- 유사 제품의 기능·수치·리뷰·인증·사이즈·카피를 현재 제품 사실로 옮기지 않는다.
- 제품 길이·구성에 관한 후기나 충돌이 발견되면 실측·구성 확인 우선순위를 높이되, 후기 내용을 보편 사실이나 광고 카피로 쓰지 않는다.
- 제3자의 제품, 카피, 로고, 고유 그래픽과 상세페이지를 복제하지 않는다.

## 5. 정보 단위와 가변 장수 강제 계약

먼저 중복을 제거한 정보 원장을 만든다.

```text
INFO_ID | 구매 질문 | 독점 Fact ID | 필수 시각 증거 | 다른 장과 겹치지 않는 새 정보 | 상태
```

`plan-gate.md`에는 아래 항목이 반드시 있어야 한다.

```text
정보 단위 수: <3~10>
선정 장수: <3~10>
장수 결정 근거: <왜 이 장수면 충분한가>
삭제·병합 역할: <비교·추천·요약 등 없앤 반복 역할>
장수 결정 상태: 완료
```

강제 규칙:

1. `정보 단위 수`와 `선정 장수`는 같아야 한다.
2. 01은 `PROBLEM_HOOK`이다. 고객의 실제 불편과 제품이 제안하는 답을 연결한다.
3. 전체에 `PRODUCT_INTRO`는 정확히 한 장만 둔다. 다른 장의 결론을 장점 목록으로 미리 반복하지 않는다.
4. 중간 장은 상품에 실제로 필요한 역할만 선택한다. 예: `FEATURE_EVIDENCE`, `FIT_STRUCTURE`, `MEASURED_SIZE`, `COMPONENTS`, `OPTIONS`, `HOW_TO_USE`, `CARE_GUIDE`, `SAFETY_GUIDE`, `SITUATION_USE`.
5. 마지막 장은 마지막 독점 정보를 제공한 뒤 중립 CTA를 붙인다. 새 정보가 없는 독립 CTA 장은 만들지 않는다.
6. `NEED_REASON`, `NEUTRAL_COMPARE`, `RECOMMENDATION`, `USE_CASES`, 독립 `CTA`는 새 독점 사실이 있을 때만 허용한다.
7. 한 장을 삭제해도 구매자가 잃는 새 정보가 없으면 그 장을 삭제한다.
8. 각 장의 `INFO_ID / 고유 구매 질문 / PRIMARY_FACT / PROOF_ID / SHOT_ID / SCENE_ID / LAYOUT_ID`는 재사용하지 않는다. Fact·Proof가 다른 정보 장은 같은 ROLE_ID를 쓸 수 있지만 PRODUCT_INTRO는 한 번만 쓴다.
9. 제품명·법정 주의문 외 동일한 4어절 이상 카피와 같은 구매 논리를 다른 장에서 반복하지 않는다.
10. 실측·판매 구성처럼 구매 결정에 직접 필요한 미확인 정보가 있으면 페이지를 꾸며 채우지 말고 질문을 한 번에 하나씩 받아 확정한다.

## 6. 카피 매니페스트

- 모든 문구는 자연스러운 한글이다.
- 한 이미지의 모든 문자열을 `EYEBROW / H1 / BODY / CARD·CHIP / CAPTION / CTA`로 먼저 고정한다.
- 큰 `H1`은 한 장에 하나만 사용한다. 목적에 맞는 설명·카드·디테일 캡션은 충분히 넣는다.
- 고객 불편 또는 검증된 제품 차이에서 짧고 구체적인 후킹을 만든다.
- 01은 제품 사양을 나열하기보다 실제 불편을 먼저 보여준다. `이런 경험 있으셨나요?`는 선택 예시이지 고정 문구가 아니다.
- 같은 훅 논리를 문제형·비교형·추천형으로 바꿔 반복하지 않는다.
- 승인 문구를 생성 중 축약·의역·추가하지 않는다.

## 7. 타이포그래피와 디자인

- 승인된 모든 장은 같은 인상의 현대적인 한국어 산세리프 한 계열을 사용한다.
- 800px 원본 기준 `H1 72~104px`, `H2 44~64px`, `BODY 32~40px`, `라벨·캡션 최소 32px`를 기본 토큰으로 사용한다.
- 400 CSS px로 축소해도 모든 정보 문구가 16px 이상이 되게 한다.
- 굵기는 `H1 800~900`, `H2·카드 700~800`, `BODY 500~600`, `라벨·캡션 600 이상`으로 유지한다.
- 밝은 배경에는 짙은 글자, 사진에는 충분한 단색 여백이나 고대비 오버레이를 둔다.
- 실제 레퍼런스는 여백, 색상 관계, 정보 위계와 촬영 문법만 참고한다.
- 상품 원본의 형태, 색상, 구성품, 라벨과 부품을 레퍼런스 스타일에 맞춰 바꾸지 않는다.

## 8. 광고 제한

- 입력하지 않은 가격, 할인율, 리뷰, 별점, 판매량, 랭킹, 인증, 시험, 수상, 보증과 환불 문구를 만들지 않는다.
- 물성·성능·내구·효능의 실증 카피는 실제 동일 제품, 기록된 조건과 연속 원본이 있는 `REAL_TEST`에만 연결한다.
- `REAL_DEMO`는 실제 제품의 보이는 구조·착용·조작만 설명한다.
- 생성형 영상과 ComfyUI 결과는 `AI_ILLUSTRATION`이며 실제 시험 증거가 아니다.
- 치료·개선 보장, 근거 없는 `100%`, `완벽`, `무조건`, `최고`, `1위`, 신체 변화형 Before/After를 금지한다.
- 쿠팡 로고, 앱 UI, 검색·장바구니·하트 아이콘과 스마트폰 목업을 넣지 않는다.
- 이미지 픽셀 안에 Page, 장 번호와 페이지 번호를 넣지 않는다.

## 9. 생성 순서

1. 입력 이미지와 상품 정보를 확인한다.
2. 동일·유사 제품을 조사하고 근거 범위를 나눈다.
3. 사실·금지 원장과 기능 우선 게이트를 만든다.
4. 독점 정보 단위를 만들고 반복 역할을 삭제·병합해 3~10장 중 최소 충분 장수를 정한다.
5. 실측·구성 충돌이 있으면 한 응답에 질문 하나, 번호 보기 마지막에 `다른 답변하기`를 두고 확인한다.
6. 선정된 모든 장의 카피·장면·읽는 순서·타이포그래피를 기획한다.
7. `asset-map.md`에 Asset Registry·Evidence Ledger·Reference Principle Ledger·Page Source Contract를 작성한다.
8. `validate_plan.py`, `validate_asset_map.py`, 의미 중복 직접 검사와 `validate_motion.py`를 통과한다.
9. `build_plan_review.py`로 선정 장수의 주제·카피·화면 구성·RAW·Proof·레퍼런스 원리·주장 경계·모션 후보가 담긴 `plan-review.html`을 만든다.
10. HTML을 보여주고 사용자 번호 승인을 받으며 HTML의 소스 해시를 승인 기록에 복사한다. 승인 전에는 이미지·GIF·영상을 생성하지 않는다.
11. 기획 승인 뒤에만 실사 촬영·ComfyUI·인계·정적 전용 중 제작 범위를 한 질문으로 확인한다.
12. 최종 생성 승인을 기록하고 `validate_generation_gate.py`를 통과한 대상만 실행한다.
13. 정적 게이트 통과 뒤 선정된 각 장을 별도 `image_gen` 호출로 제품·디자인·승인 한국어까지 한 번에 완성한다.
14. 텍스트, 제품, 레이아웃, 광고 주장, 자산 계보, 정보 독점과 전체 흐름을 직접 검사하고 실패한 장만 다시 생성한다.
15. `page-01.png`부터 선정된 마지막 장까지 연속 번호, 모두 `800x2400`인지 검증·정규화한다.

```text
장면과 구도: 800x2400, 1:3 세로, 3~6개 정보 구역과 명확한 읽는 순서
카피 직접 조판: 승인 문자열을 정확히 렌더링하고 자리표시자를 남기지 않음
카피 매니페스트: EYEBROW "..." / H1 "..." / BODY "..." / CARD·CHIP·CAPTION ["..."] / CTA "..."
금지: 매니페스트 밖 카피, 깨진 한글, 32px 미만 문구, 더미 영문, 허위 주장, 워터마크, 쿠팡 UI, 레퍼런스 복제
```

## 11. QA 기준

- 승인 카피와 글자 단위로 일치하고, 자모 깨짐·오탈자·추가 문구가 없다.
- 기능적 구매 이유가 디자인보다 먼저 읽히며 기능 소유 장에서 근거 범위가 분명하다.
- 각 장이 독점 `INFO_ID`, 구매 질문, 사실과 시각 증거를 실제로 전달한다.
- 장의 목적에 맞는 3~6개 정보 구역이 있고 같은 내용을 바꿔 말하지 않는다.
- 전체 장의 서체, 크기 관계, 팔레트와 무드는 일관되지만 제품 컷·장면·지배 레이아웃은 구분된다.
- 제품 실루엣, 비율, 색상, 부품·라벨과 확인된 구성 수가 원본과 맞는다.
- 실측·수량·기능 수치가 허용 근거와 정확히 일치한다.
- 생성 장면·WEB_MATCH·REF_*를 현재 상품 원본이나 실제 시험처럼 표현하지 않고 승인한 자산 계보와 일치한다.
- 허위 사실, 금지 UI, 로고, 페이지 번호와 근거 없는 효과 암시가 없다.
- 한 장을 가렸을 때 사라지는 독점 구매 정보가 없다면 `INFORMATION_GAIN=RETRY`로 판정하고 삭제·병합한다.
- 최종 파일은 `page-01`부터 선정된 마지막 번호까지 빠짐없이 연속하고 모두 `800x2400`이다.

실패한 장만 다시 생성한다. 텍스트 실패는 먼저 `image_gen` 직접 조판을 재시도하며, 후편집은 사용자 선택을 받은 예외 경로다.
