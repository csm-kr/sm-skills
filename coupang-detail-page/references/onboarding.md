# 첫 실행 온보딩

## 목적

처음 사용하는 사람도 원본 상품, 실측·실사 근거, 웹 확인 보조와 실제 상세페이지 레퍼런스를 구분하고, 확인된 정보만으로 서로 다른 구매 질문에 답하는 상세페이지를 만들도록 안내한다. 장수는 먼저 정하지 않는다. 고유한 정보 단위를 확인한 뒤 `3~10장`, 보통 `5~8장` 안에서 필요한 만큼만 선택한다.

## 절대 대화 규칙

1. 한 번의 응답에는 질문을 정확히 하나만 넣는다.
2. 선택 질문은 번호 보기로 제시하고 마지막 보기는 항상 `다른 답변하기`로 쓴다.
3. 사용자가 번호 하나만 답해도 진행할 수 있게 한다. `다른 답변하기`를 고르면 다음 응답에서 자유 입력 하나만 요청한다.
4. 아직 묻지 않은 다음 질문을 같은 응답에 미리 붙이지 않는다.
5. 답을 받으면 저장한 값 하나를 확인하고 다음 질문 하나만 한다.
6. 파일, 사진, 웹 리서치와 이전 답변에서 이미 확인한 사실은 다시 묻지 않는다.
7. 사진에서 관찰 가능한 외형과 사용자가 확인한 사실을 구분한다. 기능·소재·치수·효능을 사진 비율로 추정하지 않는다.
8. 기획 HTML 승인 전에는 `image_gen`, GIF·영상 또는 ComfyUI 렌더를 호출하지 않는다.
9. 입력 확인, 웹 조사, 정보 단위 선정, 기획, 생성, QA와 납품 관문마다 `완료한 것 / 현재 하는 것 / 다음 할 것`을 짧게 중계한다.

## 시작 설명

첫 질문 전에 다음을 짧게 한 번 설명한다.

- 원본 상품 사진은 `<skill-root>/inputs/<project-no>/original-images/`에 저장한다. 최소 1장, 정면·측면·디테일·구성품·착용/사용 모습을 포함한 3~8장을 권장한다.
- 실제 상세페이지 레퍼런스는 `<skill-root>/inputs/<project-no>/real-references/`에 저장한다. 1~3장을 권장하지만 선택 사항이며 레이아웃·색감·여백만 참고한다.
- 상품 정보는 `<skill-root>/inputs/<project-no>/product-info.md`에 답변마다 저장한다.
- 동일 제품과 유사 제품을 웹에서 찾고, 출처·일치 수준·사실 확인 범위와 기획 전용 패턴을 분리해 기록한다.
- 가장 중요한 기능적 구매 이유를 디자인보다 먼저 확인한다. 그 뒤 실측, 사양, 구성품, 사용법과 관리법처럼 실제 구매 판단에 새 정보를 주는 항목을 우선한다.
- 장수는 고정하지 않는다. `고유 INFO_ID 하나 = 구매 질문 하나 = 페이지 하나`로 설계하고, 반복되는 비교·추천·활용·요약·독립 CTA는 삭제하거나 병합한다.
- 첫 페이지는 `PROBLEM_HOOK`, 상품 소개는 정확히 한 번만 `PRODUCT_INTRO`로 둔다. 마지막 정보 페이지에 중립 CTA를 함께 넣어 CTA만을 위한 페이지는 만들지 않는다.
- 기획이 끝나면 `plan-review.html`에서 선정 장수, 장수 근거, 각 장의 고유 질문·INFO_ID·카피·화면 구성·근거와 모션 후보를 보여주고 확정받는다.
- 승인된 각 페이지는 `image_gen` 한 번의 호출에서 제품 장면과 정확한 한글 카피를 함께 조판한다.
- 모든 페이지는 같은 한국어 산세리프 계열, 큰 헤드라인, 충분한 대비와 여백을 공유한다. 800px 원본에서 본문·라벨·캡션은 최소 32px로 둔다.
- 기본 규격은 `800x2400`이며, 결과는 `page-01.png`부터 선정한 마지막 페이지 번호까지 연속 저장한다.
- 진행 상태는 `입력 확인 → 웹 조사 → 정보 단위·장수 결정 → 기획 HTML 승인 → N/선정 장수 생성·검수 → 납품`으로 알린다.

## 질문 순서

프로젝트 번호를 확정하면 `python3 scripts/init_project.py --prepare <project-no>`를 실행한다. 이미 작성된 입력은 보존한다.

### 1. 프로젝트 선택

```text
1번 질문. 어느 프로젝트로 시작할까요?
1. 현재 시작 프로젝트 계속 사용
2. 새 번호 프로젝트 생성
3. 기존 프로젝트 번호 입력
4. 다른 답변하기
번호로 답해주세요.
```

사용자가 이미 프로젝트 번호를 지정했다면 건너뛴다.

### 2. 원본 상품 사진

폴더와 대화 첨부를 먼저 검사한다. 대화 첨부에 로컬 경로가 있으면 원본 바이트를 번호 프로젝트에 보존하고, 경로가 없으면 저장이 필요하다고 안내한다. 이미지가 있으면 다음처럼 묻는다.

```text
2번 질문. 현재 원본 사진으로 진행할까요?
1. 그대로 진행
2. 사진을 더 추가
3. 저장 위치를 다시 안내
4. 다른 답변하기
번호로 답해주세요.
```

이미지가 없으면 다음처럼 묻는다.

```text
2번 질문. 원본 상품 사진을 어떻게 넣을까요?
1. 지금 대화에 첨부
2. 저장 폴더 위치 다시 보기
3. 작업 중단
4. 다른 답변하기
번호로 답해주세요.
```

원본 없이 생성하는 선택지는 주지 않는다. 대화 첨부는 최대 5장까지 입력 후보로 검사할 수 있지만, HTML 승인 전 모두 `inputs/<project-no>/`의 역할별 폴더에 원본 바이트로 저장하고 `asset-map.md`에 실제 SHA256을 기록해야 한다. 저장·해시 전 첨부는 기획 증거, 승인 대상 또는 생성 입력이 아니다. `--conversation-originals`는 미저장 상태를 드러내는 진단 옵션이며 생성 허가가 아니다.

### 3. 실제 상세페이지 레퍼런스

```text
3번 질문. 참고 상세페이지를 어떻게 할까요?
1. 넣어둔 레퍼런스로 진행
2. 레퍼런스 1~3장 추가
3. 레퍼런스 없이 기본 디자인으로 진행
4. 다른 답변하기
번호로 답해주세요.
```

레퍼런스의 상품, 로고, 고유 카피, 수치와 식별 가능한 디자인은 복제하지 않는다.

### 4. 상품 정보 입력 방식

```text
4번 질문. 상품 정보를 어떻게 입력할까요?
1. 한 항목씩 질문받기
2. 작성한 product-info.md 사용
3. 정리된 상품 설명을 한 번에 붙여넣기
4. 다른 답변하기
번호로 답해주세요.
```

한 항목씩 입력할 때는 다음 항목을 한 턴에 하나씩 묻는다.

1. 상품명
2. 카테고리
3. 가장 중요한 기능적 구매 이유
4. 그 기능이 해결하는 대표 불편
5. 그 기능의 사용자 확인·수치·인증·시험·실물 근거
6. 검증된 핵심 장점 한 가지씩
7. 타깃 고객
8. 대표 사용 상황
9. 사용 방법
10. 판매 구성품
11. 판매 구성 수량
12. 소재·재질
13. 색상
14. 현재 판매 옵션
15. 관리·세탁·보관 방법
16. 주의사항
17. 경쟁 제품 대비 검증된 차별점
18. 객관적인 수치·인증·시험 정보
19. 사용 금지 표현
20. 실물 샘플·실사 촬영 가능 여부

하나의 응답에서 위 항목 여러 개를 동시에 요구하지 않는다. 입력이 없는 사실을 채우기 위해 유사 제품 정보를 전용하지 않는다.

기능 근거를 확인할 때 정성적 불편·구조는 사용자가 확인했고 실제 비교 사진만 없다면, 사진 추가를 기본으로 요구하지 않는다. `현재 RAW + 사용자 확인 + 정적 AI_ILLUSTRATION`으로 기록하고 다음 질문으로 진행한다. AI 비교 장면은 기획 HTML 승인과 정적 생성 게이트 뒤 `image_gen`으로 만들며 실제 비교 시험·수치·객관적 우위의 증거가 아니다. 사용자가 실증 비교를 명시적으로 원하거나 물성·성능 주장이 필요한 경우에만 실제 비교 촬영을 한 질문으로 요청한다.

### 5. 실측 정보 확인

실측은 독립적인 구매 질문에 답할 수 있는 우선 정보다. 현재 제품 자체를 자로 잰 사용자 값·자와 함께 찍은 현재 제품 사진, 또는 정확한 동일 옵션의 `M1 + E1/E2` 공식 치수만 허용한다. 사진 비율, 유사 제품, 리뷰, 모델의 추정값으로 치수를 만들지 않는다.

먼저 이 질문 하나만 한다.

```text
실측 질문. 현재 상품의 실측 정보를 어떤 방식으로 확인할까요?
1. 자로 잰 cm 값을 한 항목씩 입력
2. 현재 상품을 자와 함께 찍은 사진 추가
3. 정확한 동일 옵션의 공식 치수 자료 사용
4. 실측 페이지 제외
5. 다른 답변하기
번호로 답해주세요.
```

- `1`이면 다음 응답에서 `전체 길이` 하나만 묻는다. 그 값을 저장한 뒤 상품에 맞는 `윗부분 평폭`, `가장 넓은 평폭`, `아랫부분 평폭`, `손목·커프 길이`, `두께·깊이` 중 필요한 다음 치수 하나씩만 묻는다.
- `2`이면 다음 응답에서 사진 한 종류만 요청하고, 자의 시작점·끝점이 보이지 않으면 해당 치수를 확정하지 않는다.
- `3`이면 다음 응답에서 직접 URL 또는 파일 하나만 요청한다. 정확한 모델·옵션 일치가 확인되지 않으면 사용하지 않는다.
- `4`이면 `실측 미제공`으로 기록하고 측정 페이지를 만들지 않는다. 빈자리를 반복 카피로 채우지 않는다.

값 입력 예시는 질문마다 하나만 제공한다.

```text
전체 길이를 cm로 알려주세요. 예: 38.5
```

### 6. 필수 웹 리서치

상품명·카테고리·원본 외형이 확보되면 별도 허락을 다시 묻지 않고 동일 제품과 유사 제품을 검색한다. [research-motion.md](research-motion.md)의 `M0~M3`, `E1~E4`를 적용한다.

- 동일 제품은 라벨, 모델명, 패키지 문구와 독특한 외형을 조합해 찾는다.
- 유사 제품은 고객 질문, 정보 모듈, 장면과 시각 문법에만 사용한다.
- 동일 제품이라도 옵션이나 원본과 충돌하면 임의로 합치지 않는다.
- 실측은 정확한 동일 옵션의 공식 자료가 아니면 상품 사실로 사용하지 않는다.
- 결과가 없으면 검색어와 `M0`를 기록한다. 인터넷 접근이 없으면 조사했다고 가장하지 않는다.

### 7. 정보 단위와 가변 장수 결정

확정한 사실을 다음 기준으로 묶는다.

```text
INFO_ID | 구매 질문 | 독점 Fact ID | 필수 시각 증거 | 다른 장에 없는 새 정보 | 상태
```

- 서로 다른 구매 질문에 답하는 정보 단위만 페이지 후보가 된다.
- 같은 사실을 문제·비교·추천·활용·요약 말투로 바꾼 후보는 하나로 병합한다.
- 실측, 검증 사양, 구성품, 사용법과 관리·주의처럼 구매 판단을 바꾸는 정보는 우선한다.
- `3~10장`, 보통 `5~8장` 안에서 `정보 단위 수 = 선정 장수`로 정한다. 정보가 부족하면 3~4장으로 줄이고, 장수를 맞추기 위해 내용을 발명하지 않는다.
- 첫 페이지는 `PROBLEM_HOOK`, `PRODUCT_INTRO`는 정확히 하나다. 마지막 정보 단위에 CTA를 합친다.

`plan-gate.md`에는 `정보 단위 수`, `선정 장수`, `장수 결정 근거`, `삭제·병합 역할`, `장수 결정 상태`를 기록한다.

### 8. 기획 HTML 검토

`check_project.py`, `validate_plan.py`, `validate_asset_map.py`, `validate_motion.py`를 통과한 뒤 `build_plan_review.py`를 실행한다. HTML에는 선정한 모든 페이지의 RAW·Proof·레퍼런스 원리·주장 경계와 모션 후보를 배치한다.

```text
기획 검토 질문. plan-review.html의 선정 장수와 각 페이지 구성을 확정할까요?
1. 이 기획 확정
2. 수정 후 다시 보기
3. 작업 중단
4. 다른 답변하기
번호로 답해주세요.
```

승인된 경우에만 `generation-gate.md`에 응답과 날짜, HTML의 `data-source-digest` 64자와 승인한 `plan-review.html` 파일 자체의 SHA256을 기록한다. 둘 중 하나라도 비거나 현재 파일과 다르면 승인되지 않은 것이다. 카피, INFO_ID, 장수, 화면 근거, 폰트 또는 모션 기획이 바뀌면 두 해시와 모든 승인을 무효화하고 HTML을 다시 검토받는다.

### 9. 승인 뒤 정확한 모션 ID 선택

```text
실행할 모션 후보를 선택해주세요.
1. motion-01 · <CAPTURE_MODE> · <GIF 또는 VIDEO>
2. motion-02 · <CAPTURE_MODE> · <GIF 또는 VIDEO>
3. motion-01 + motion-02 · <추천 조합>
4. 모션 없이 정적 이미지만
5. 다른 답변하기
번호로 답해주세요.
```

보기는 현재 `motion-plan.md`에 존재하는 정확한 `motion-NN`, CAPTURE_MODE와 지원 형식으로 다시 작성한다. 선택한 ID를 중복 없는 쉼표 목록으로만 `선정 모션 ID`에 기록하고 후보명·설명을 섞지 않는다. 정적 전용은 `해당 없음`으로 기록한다. 사용자가 다른 조합을 원하면 다음 응답에서 정확한 ID 목록 하나만 묻는다.

### 10. 정확한 제작 범위 선택

선정 ID가 선언한 형식과 호환되는 보기만 남겨 질문한다.

```text
선정한 모션 ID의 제작 범위를 선택해주세요.
1. 정적 + GIF · STATIC_PLUS_GIF
2. 정적 + VIDEO · STATIC_PLUS_VIDEO
3. 정적 + GIF + VIDEO · STATIC_PLUS_GIF_VIDEO
4. 정적 + 선정 모션 제작안 인계 · STATIC_PLUS_MOTION_HANDOFF
5. 모션 ID 다시 선택
6. 다른 답변하기
번호로 답해주세요.
```

GIF 전용 후보에 VIDEO 범위를, VIDEO 전용 후보에 GIF 범위를 제시하지 않는다. 정적 전용은 이 질문을 반복하지 않고 `STATIC_ONLY`를 기록한다. 핸드오프도 정확한 모션 ID가 필요하다.

### 11. CAPTURE_MODE에 맞는 환경 선택

- 선택 ID가 `REAL_TEST`·`REAL_DEMO`뿐이면 `1. 실제 제품 촬영·ComfyUI 불필요(NOT_REQUIRED) / 2. 촬영 원본을 live 연결된 ComfyUI로 후처리(CONNECTED) / 3. 검증된 워크플로와 촬영·편집안 인계(WORKFLOW_PROVIDED) / 4. 범위 다시 선택 / 5. 다른 답변하기`를 한 질문으로 제시한다. 3번은 실제 실행이 아니라 핸드오프다.
- `AI_ILLUSTRATION`이 하나라도 포함되면 `1. live 연결 확인된 ComfyUI에서 실제 생성(CONNECTED) / 2. 검증된 워크플로와 제작안 인계(WORKFLOW_PROVIDED) / 3. 워크플로 없이 제작안 인계(HANDOFF_ONLY) / 4. ID·범위 다시 선택 / 5. 다른 답변하기`를 한 질문으로 제시한다. 실제 AI 생성에는 `CONNECTED`만 허용한다.
- `STATIC_ONLY`는 `NOT_REQUIRED`, `STATIC_PLUS_MOTION_HANDOFF`는 검증된 워크플로를 함께 넘길 때 `WORKFLOW_PROVIDED`, 그렇지 않으면 `HANDOFF_ONLY`로 기록한다.
- 실제 응답은 `YYYY-MM-DD · SELECT_ENV · SCOPE=<범위> · MOTION=<정렬 ID 또는 NONE> · COMFY=<상태>` 형식으로 `환경 선택 기록`에 남긴다. 승인 날짜는 Asia/Seoul 오늘보다 미래일 수 없고 `기획 승인 ≤ 환경 선택 ≤ 최종 승인` 순서여야 한다. `CONNECTED`·`WORKFLOW_PROVIDED`는 [comfyui-receipt-schema.md](comfyui-receipt-schema.md)에 따라 `scripts/comfyui_receipt.py`가 만든 receipt 경로를 기록한다. helper 계약에 맞지 않는 수기·generic JSON은 무효다. CONNECTED는 24시간 이내 receipt와 생성 게이트 시점 live 재확인을 모두 통과해야 하며 WORKFLOW_PROVIDED는 실제 렌더가 아닌 핸드오프 전용이다.

실물·연결이 없으면 생성했다고 보고하지 않는다. `AI_ILLUSTRATION`은 실증 자료가 아니며 효능·성능을 입증하는 카피를 붙이지 않는다. REAL_TEST·REAL_DEMO 실제 GIF·영상 실행은 FFmpeg의 `ffprobe`가 필수이며 RAW_TEST·RAW_DEMO 원본의 decode 가능 여부, dimensions, duration, frames를 검사한다. `ffprobe`가 없거나 어느 항목이든 확인되지 않으면 모션은 fail-closed한다.

### 12. 정적 이미지 승인

```text
정적 상세페이지 이미지 생성을 승인할까요?
1. 승인된 기획대로 정적 이미지 생성 승인
2. 기획 HTML 다시 보기
3. 제작 범위 다시 선택
4. 작업 중단
5. 다른 답변하기
번호로 답해주세요.
```

`1`을 받은 뒤에만 `정적 이미지 생성 승인: 승인`으로 기록한다.

### 13. GIF·영상 또는 핸드오프 승인

모션 범위일 때만 정확한 ID·형식·CAPTURE_MODE를 한 줄로 요약한 뒤 질문 하나를 한다.

```text
선정한 GIF·영상 범위를 승인할까요?
1. 선정 모션의 실제 생성 승인
2. 정적 이미지만 진행
3. 모션 ID·형식·환경 다시 선택
4. 작업 중단
5. 다른 답변하기
번호로 답해주세요.
```

핸드오프 범위에서는 1번을 `선정 모션 제작안 인계 준비 승인`으로 바꾼다. 두 경우 모두 승인 시 `GIF·영상 생성 승인: 승인`을 기록하지만 HANDOFF_ONLY의 승인은 실제 렌더 권한이 아니다. 정적 전용은 `GIF·영상 생성 승인: 해당 없음`으로 기록한다.

## 생성 전 마지막 확인

제작 범위, 정확한 모션 ID, CAPTURE_MODE, GIF·VIDEO 형식, ComfyUI 상태와 실제 실행 대상을 한 줄로 요약한 뒤 다음 질문 하나만 한다.

```text
마지막 질문. 확정한 범위의 실제 생성을 시작할까요?
1. 생성 시작
2. 기획 HTML 다시 보기
3. 제작 범위 다시 선택
4. 작업 중단
5. 다른 답변하기
번호로 답해주세요.
```

`1`을 받은 뒤 `YYYY-MM-DD · APPROVE_EXECUTION · SCOPE=<범위> · MOTION=<정렬 ID 또는 NONE> · COMFY=<상태>` 형식으로 `최종 생성 승인 기록`에 남긴다. 자유문·거절문·현재 범위·ID·환경과 모순된 기록은 무효이며 환경을 바꾸면 최종 승인을 다시 받는다. 이 값은 앞의 정적·모션 개별 승인을 대신하지 않는다. 승인 뒤에만 정적 또는 모션 생성 게이트를 통과시킨다. 선정한 페이지마다 `image_gen`을 한 번씩 호출하며 `PAGE_NO`, `ROLE_ID`, 고유 `INFO_ID`, 구매 질문, 필수 시각 증거와 승인 카피를 반복한다. 승인된 `prompt-set.md`는 그대로 잠그고 호출·재시도·실패·선택본은 `execution-log.md`, 장별 최종 판정은 `qa-report.md`에 기록한다. 생성 진행은 `N/선정 장수`로 알린다.

각 결과를 직접 확대 검수해 크기, 한글, 제품 정체성, INFO_ID 역할과 비중복을 확인한다. 정적 이미지 규격 검사는 Pillow 자동 runtime을 사용하고 안전한 자동 대체 경로를 적용하되, 결과를 다시 시각 검사한다. 실패한 페이지만 같은 디자인 시스템과 원본으로 다시 생성한다.

정적 한글은 승인 카피를 포함한 `image_gen` 직접 생성이 1차 경로다. 직접 텍스트 시도 2회가 모두 실패한 뒤에만 `1. 승인 카피 FIXED_TEXT_LAYER 후편집 허용 / 2. image_gen 한 번 더 생성 / 3. 카피 단순화 후 재승인 / 4. 다른 답변하기`를 한 질문으로 묻는다. 사용자가 1번을 명시적으로 승인하기 전에는 정적 고정 레이어로 대체하지 않는다. GIF·영상 광고 카피는 처음부터 실제 폰트의 고정 텍스트 레이어를 사용한다.
