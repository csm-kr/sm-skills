# 첫 실행 온보딩

## 목적

처음 사용하는 사람이 폴더 구조를 몰라도 원본 상품, 스타일 레퍼런스와 웹 리서치 자료를 혼동하지 않고, 검증된 정보와 직접 조판한 한글 카피로 상세페이지 10장을 만들도록 안내한다.

## 절대 대화 규칙

1. 한 번의 응답에는 질문을 정확히 하나만 넣는다.
2. 선택 질문은 `1.`, `2.`, `3.` 형식의 번호 보기로 제시하고, 마지막 보기는 개수와 관계없이 항상 `다른 답변하기`로 쓴다.
3. 사용자가 번호 하나만 답해도 진행할 수 있게 한다. `다른 답변하기`를 고르면 같은 턴에 추가 질문을 붙이지 말고 다음 응답에서 자유 입력 하나만 요청한다.
4. 자유 입력 질문에는 답변 형식 예시 하나만 준다. 예: `살랑 팔토시`.
5. 아직 묻지 않은 다음 질문을 같은 응답에 미리 붙이지 않는다.
6. 답을 받으면 저장한 값 하나를 확인하고 다음 질문 하나만 한다.
7. 파일, 사진, 웹 리서치와 이전 답변에서 이미 확인한 사실은 다시 묻지 않는다.
8. 사진으로 보이는 속성도 확정 사실로 저장하기 전에 사용자 확인 또는 충분한 출처 등급을 확보한다.
9. 생성 준비가 끝나기 전에는 `image_gen`을 호출하지 않는다.
10. 제안값이 틀릴 수 있음을 전제로, 사용자가 모든 선택 질문에서 `다른 답변하기`로 제안을 벗어날 수 있게 한다.
11. 입력 확인, 웹 조사, 기획, 생성, QA, 납품 관문마다 `완료한 것 / 현재 하는 것 / 다음 할 것`을 짧게 중계하고 업데이트 없는 상태가 60초를 넘지 않게 한다.

## 시작 설명

첫 질문 앞에서 다음 내용을 짧게 한 번 설명한다.

- 원본 상품 사진은 `<skill-root>/inputs/<project-no>/original-images/`에 저장한다. 최소 1장, 정면·측면·디테일·구성품·착용/사용 모습을 포함한 3~8장을 권장한다.
- 참고할 실제 상세페이지 이미지는 `<skill-root>/inputs/<project-no>/real-references/`에 저장한다. 1~3장을 권장하지만 선택 사항이다. 레이아웃·색감·여백만 참고하며 그 상품, 로고, 문구와 수치는 복제하지 않는다.
- 상품 정보는 `<skill-root>/inputs/<project-no>/product-info.md`에 답변마다 저장한다.
- 동일 제품과 유사 제품을 웹에서 반드시 찾고, 접근 가능한 실제 상세페이지를 최소 3개 비교해 출처·일치 수준과 후킹·정보 모듈 패턴을 기록한다. 동일 제품 자료만 등급에 따라 사실 확인을 보조할 수 있고, 유사 제품은 기획 아이디어에만 사용한다.
- `prompt.txt`의 10장 설득 흐름을 기준으로 정확한 한글 카피와 디자인을 먼저 기획한다. 헤드라인만 반복하지 않고 설명, 카드, 디테일 캡션, 추천 라벨과 CTA를 장에 맞게 3~6개 정보 구역으로 구성한다.
- 각 장은 `image_gen` 한 번의 호출에서 제품 이미지와 승인된 카피 매니페스트를 함께 직접 조판한다.
- 10장은 같은 한국어 산세리프 계열, 큰 후킹 헤드라인, 충분한 대비와 여백을 공유한다. 800px 원본에서 본문·라벨·캡션은 최소 32px로 두어 모바일 16px 상당을 확보한다.
- 후보는 `outputs/<project-no>/raw/`, 최종본은 `outputs/<project-no>/final/`에 `page-01.png`부터 `page-10.png`로 저장한다.
- 기본 최종 규격은 모두 `800x2400`이다. 생성 후 직접 확대 검수하고, 깨진 한글·잘못된 제품·낮은 가독성이 있는 장만 다시 생성한다.
- 정지 이미지 기획과 함께 GIF·영상으로 효과적인 장면을 제안할 수 있으며, 사용자가 원하면 선택적인 ComfyUI 핸드오프 자료도 준비한다.
- 입력에 없는 효능, 가격, 리뷰, 인증과 수치는 만들지 않으며 준비 검사를 통과해야 생성한다.
- 진행 중에는 `입력 확인 → 웹 조사 → 10장 기획 → N/10 생성·검수 → 정규화·납품` 현황을 중간중간 알려준다.

설명 뒤에는 아래 순서의 현재 질문 하나만 보인다.

## 질문 순서

프로젝트 번호를 확정한 즉시 `python3 scripts/init_project.py --prepare <project-no>`를 실행한다. 이미 작성된 `product-info.md`는 보존되며, 새 설치에서는 Git에서 제외되는 런타임 양식이 생성된다.

### 1. 프로젝트 선택

기존 빈 시작 프로젝트가 있으면 다음처럼 묻는다.

```text
1번 질문. 어느 프로젝트로 시작할까요?
1. 현재 001 계속 사용
2. 새 번호 프로젝트 생성
3. 기존 프로젝트 번호 입력
4. 다른 답변하기
번호로 답해주세요.
```

사용자가 이미 프로젝트 번호를 지정했다면 이 질문은 건너뛴다.

### 2. 원본 상품 사진

먼저 폴더와 대화 첨부를 검사한다. 이미지가 있으면 개수와 관찰 가능한 시점만 말하고 다음 중 하나를 묻는다.

```text
2번 질문. 현재 원본 사진으로 진행할까요?
1. 그대로 진행
2. 사진을 더 추가
3. 저장 위치를 다시 안내
4. 다른 답변하기
번호로 답해주세요.
```

이미지가 없으면 다음처럼 묻는다.

```text
2번 질문. 원본 상품 사진을 어떻게 넣을까요?
1. 지금 대화에 첨부
2. 저장 폴더 위치 다시 보기
3. 작업 중단
4. 다른 답변하기
번호로 답해주세요.
```

원본이 없는 상태에서 생성으로 진행하는 선택지는 주지 않는다. 대화 첨부를 로컬 파일로 복사할 수 없으면 대화에서만 사용한다고 정확히 알리고, 실제 첨부 개수를 생성 전 검사에 `--conversation-originals <count>`로 전달한다. 대화 전용 이미지는 최대 5장만 허용한다. 6장 이상이면 핵심 원본을 최대 5장으로 선별할지, 나머지를 로컬 폴더에 저장할지 한 질문으로 확인하며 마지막 보기는 `다른 답변하기`로 둔다.

### 3. 실제 상세페이지 레퍼런스

레퍼런스 개수를 확인한 뒤 다음처럼 묻는다.

```text
3번 질문. 참고 상세페이지를 어떻게 할까요?
1. 넣어둔 레퍼런스로 진행
2. 레퍼런스 1~3장 추가
3. 레퍼런스 없이 기본 디자인으로 진행
4. 다른 답변하기
번호로 답해주세요.
```

레퍼런스는 선택 사항이다. 제공된 레퍼런스뿐 아니라 필수 웹 리서치에서 찾은 카테고리 디자인도 추상적인 구성 원리만 참고할 수 있다. 어느 경우에도 특정 상세페이지의 상품, 로고, 고유 카피, 수치와 식별 가능한 디자인을 복제하지 않는다.

### 4. 상품 정보 입력 방식

```text
4번 질문. 상품 정보를 어떻게 입력할까요?
1. 한 항목씩 질문받기
2. 작성한 product-info.md 사용
3. 정리된 상품 설명을 한 번에 붙여넣기
4. 다른 답변하기
번호로 답해주세요.
```

첫 실행 기본값은 `1`이다. 사용자가 한 항목씩을 선택하면 아래 항목을 한 턴에 하나씩 묻는다. 각 자유 입력 질문에는 사용자가 그대로 답할 수 있는 예시 하나만 보여준다.

1. 상품명
2. 카테고리
3. 핵심 장점 1
4. 핵심 장점 2
5. 핵심 장점 3
6. 타깃 고객
7. 대표 불편
8. 사용 상황
9. 사용 방법
10. 구성품
11. 소재/재질
12. 색상
13. 사이즈, 중량, 제조국 중 제공 가능한 정보
14. 관리/세탁/보관 방법
15. 주의사항
16. 경쟁 제품 대비 검증된 차별점 또는 `없음`
17. 강조할 분위기
18. 객관적인 수치/인증/시험 정보 또는 `없음`
19. 사용하면 안 되는 표현 또는 `없음`

사진에서 후보를 제안할 수 있는 항목은 다음처럼 확인한다.

```text
이 제안값을 어떻게 처리할까요?
1. 제안값 승인
2. 직접 수정
3. 정보 없음으로 기록
4. 다른 답변하기
번호로 답해주세요.
```

사진에서 확인할 수 없는 기능, 소재, 크기, 효능과 사용법은 제안하지 않는다.

### 5. 필수 웹 리서치

상품명·카테고리·원본 외형이 확보되면 사용자에게 별도 허락을 다시 묻지 않고 동일 제품과 유사 제품을 모두 검색한다. [research-motion.md](research-motion.md)의 `M0~M3` 동일성 등급과 `E1~E4` 출처 등급을 적용하고 직접 URL, 확인일, 관찰한 근거와 사용 가능 범위를 기록한다.

- 동일 제품 검색: 상품명, 라벨, 모델명, 패키지 문구와 독특한 외형을 조합해 찾는다.
- 유사 제품 검색: 같은 카테고리·타깃·사용 상황의 상세페이지와 시각 표현을 찾는다.
- 동일 제품이라도 원본과 충돌하면 임의로 합치지 않는다.
- 유사 제품의 사양, 효능, 수치, 카피와 제품 외형은 상품 사실로 승격하지 않는다.
- 검색 결과가 없으면 실행한 검색어와 `M0` 판정을 기록한 뒤 진행할 수 있다. 인터넷 도구 자체를 사용할 수 없으면 검색을 했다고 가장하지 않고 생성 준비를 중단해 접근 가능한 환경이 필요함을 알린다.

동일 제품 후보와 사용자 원본이 충돌해 기준 선택이 필요한 경우에만 다음 질문 하나를 한다.

```text
웹에서 찾은 동일 제품 후보와 원본 사진이 다릅니다. 무엇을 기준으로 할까요?
1. 사용자 원본 사진
2. 웹 후보를 기준으로 새 원본 추가
3. 추가 근거 이미지 입력
4. 다른 답변하기
번호로 답해주세요.
```

### 6. GIF·영상 및 ComfyUI 선택

정적 10장과 별도로 효과적인 모션 후보를 먼저 1~3개 제안한 뒤 하나만 묻는다.

```text
모션 아이디어를 결과에 어떻게 포함할까요?
1. GIF 아이디어만 포함
2. 영상 아이디어와 ComfyUI 핸드오프 포함
3. 정적 이미지 10장만 진행
4. 다른 답변하기
번호로 답해주세요.
```

ComfyUI 연결이 실제로 제공되지 않으면 실행했다고 말하지 않는다. 대신 입력 스틸, 장면 지시, 길이, 프레임 속도, 반복 방식, 네거티브 프롬프트와 검수 조건을 담은 핸드오프만 작성한다.

## 답변 저장

- 각 답을 받은 직후 해당 줄 하나를 `product-info.md`에 저장한다.
- `없음`, `미제공`도 의도적인 답으로 저장할 수 있지만 상품명, 카테고리, 장점 3개, 타깃, 불편, 사용 상황·방법, 구성품, 소재, 색상과 주의사항은 임의로 `없음` 처리하지 않는다.
- 사용자가 이전 답을 바꾸면 기존 줄만 수정하고 나머지를 보존한다.
- 이미지 파일을 옮길 때 원본은 보존하고 같은 이름을 덮어쓰지 않는다.
- 웹 리서치의 검색어, 직접 URL, 일치·출처 등급, 확인 사실과 기획 전용 아이디어를 `outputs/<project-no>/web-research.md`에 분리해 기록한다.

## 생성 전 마지막 확인

모든 답변과 리서치 뒤 `python3 scripts/check_project.py <project-no>`를 실행한다. 대화에만 존재하는 원본이 있으면 `--conversation-originals <count>`를 함께 사용한다. 실패하면 누락 항목 하나만 다음 질문으로 묻는다. 성공하면 원본 수, 레퍼런스 수, 상품명, 웹 리서치 등급, 사용할 사실과 금지 표현, 10장 카피·디자인 방향, 모션 선택과 결과 위치를 짧게 요약하고 다음 하나만 묻는다.

```text
마지막 질문. 800x2400 상세페이지 10장 생성을 시작할까요?
1. 시작
2. 기획과 입력 다시 검토
3. 작업 중단
4. 다른 답변하기
번호로 답해주세요.
```

`1`을 받은 경우에만 장별 `image_gen` 10회 생성으로 넘어간다. 각 장의 승인된 카피와 디자인을 한 호출에서 직접 조판하고, 10장을 직접 검수한 뒤 실패한 장만 원본과 동일 디자인 시스템으로 다시 생성한다.
