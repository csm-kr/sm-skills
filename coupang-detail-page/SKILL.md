# 쿠팡 상세페이지 10장 생성

## 완료 계약

- 기획안이나 프롬프트만 전달하지 말고 최종 이미지 10장을 실제로 생성한다.
- built-in `image_gen`을 기본 경로로 사용하고, 서로 다른 10개 장을 각각 별도 호출로 만든다. 한 장짜리 10분할 콜라주로 대체하지 않는다.
- 정상 실행은 장별 1회씩 10회 호출한다. 실패한 장의 재시도는 추가 호출로 기록하되 최종본은 정확히 10개만 남긴다.
- 최종 파일을 `page-01.png`부터 `page-10.png`까지 저장하고 모두 실제 래스터 크기 `780x3000`으로 검증한다.
- 사용자 제공 정보와 원본 이미지에 근거한 표현만 사용한다. 프롬프트나 레퍼런스만으로 상품 사실을 추정하지 않는다.

## 필수 입력

생성 전에 다음 항목을 확보한다.

- 상품명과 카테고리
- 상품의 핵심 장점 3가지
- 타깃 고객과 대표 불편
- 사용 상황과 사용 방법
- 구성품, 소재, 색상, 크기 등 확인 가능한 사양
- 경쟁 제품 대비 검증된 차별점
- 주의사항과 사용 금지 표현
- 객관적으로 확인된 수치, 인증, 시험 정보 또는 `없음`
- 제품 외형을 확인할 수 있는 원본 상품 이미지 1장 이상
- 선택 사항: 스타일·구성 참고용 실제 상세페이지 이미지

정확한 상품 외형이 필요한데 원본 이미지가 없거나, 이미지와 입력 정보가 충돌하면 생성 전에 사용자에게 확인한다. 누락된 부가 정보는 보수적으로 생략하고 임의로 채우지 않는다.

## 첫 실행 온보딩

처음 사용하는 프로젝트이거나 `product-info.md`가 비어 있거나 사용자가 사용법 설명을 요청하면, 다른 작업보다 먼저 [references/onboarding.md](references/onboarding.md)를 끝까지 읽고 한국어 온보딩을 진행한다.

- 한 응답에서는 질문을 정확히 하나만 한다. 여러 입력 항목을 한꺼번에 묻지 않는다.
- 선택 질문에는 `1.`, `2.`, `3.` 보기만 제시하고 사용자가 번호 하나로 답할 수 있게 한다.
- 직접 입력이 필요한 질문은 질문 번호와 함께 답변 형식 한 줄을 보여준다. 예: `4. 헬로 큐트 슬리브`.
- 이미 파일이나 대화에서 확인한 내용은 다시 묻지 않는다. 사진에서 추정한 내용은 사실로 저장하기 전에 번호 선택으로 확인받는다.
- 답을 받을 때마다 해당 프로젝트의 `product-info.md` 또는 입력 폴더에 반영하고, 저장한 값 하나를 짧게 확인한 뒤 다음 질문 하나만 한다.
- 원본 상품 사진은 3~8장, 실제 상세페이지 스타일 레퍼런스는 1~3장을 권장한다. 원본은 최소 1장이 필수이고 스타일 레퍼런스는 선택 사항임을 함께 설명한다.
- 온보딩 시작 시 원본·레퍼런스·최종 결과의 정확한 저장 위치, 10회의 장별 생성, `780x3000` 최종 규격과 사실 검증 게이트를 설명한다.
- 마지막에는 `check_project.py` 결과를 보여주고 `1. 10장 생성을 시작 / 2. 입력을 다시 검토 / 3. 중단` 중 하나만 묻는다. 사용자가 `1`을 선택하고 준비 검사가 성공한 경우에만 `image_gen`을 호출한다.

## 참조 문서

- 첫 실행, 빈 프로젝트 또는 사용법 안내가 필요하면 [references/onboarding.md](references/onboarding.md)를 읽고 한 번에 한 질문씩 진행한다.
- 원본 상품 이미지가 있으면 [references/original-images.md](references/original-images.md)를 읽고 제품 불변 요소와 이미지별 역할을 기록한다.
- 실제 상세페이지 레퍼런스가 있거나 10장 구성을 설계할 때 [references/real-references.md](references/real-references.md)를 읽고 스타일 분석, 복제 금지 규칙, 10장 청사진을 적용한다.

두 이미지 그룹의 역할을 섞지 않는다. 원본 상품 이미지는 제품 사실과 정체성의 기준이고, 실제 레퍼런스는 레이아웃·색감·정보 위계의 참고 자료일 뿐이다. 충돌할 때는 검증된 상품 정보와 원본 상품 이미지를 우선한다.

## 작업 흐름

### 1. 실행 폴더를 준비한다

스킬 폴더 아래에서 세 자리 숫자 프로젝트 번호로 실행을 관리한다. `inputs/`와 `outputs/`에 존재하는 숫자 폴더의 최댓값 다음 번호를 선택한다. 첫 프로젝트는 `001`, 다음 프로젝트는 `002`로 만든다. 삭제된 중간 번호를 자동으로 재사용하지 않는다.

`<skill-root>`는 이 `SKILL.md`가 들어 있는 `coupang-detail-page` 디렉터리다. 모든 상대 경로 명령은 `<skill-root>`를 작업 디렉터리로 실행한다.

배포본에는 `001` 시작 프로젝트의 폴더와 빈 `product-info.example.md`가 포함되어 있다. 설치 직후 `<skill-root>`에서 `python3 scripts/init_project.py --prepare 1`을 한 번 실행해 Git에서 제외되는 비공개 런타임 파일 `inputs/001/product-info.md`를 만든다. 이미 있으면 내용을 보존한다. 첫 실행은 이 파일과 이미지 폴더를 채워 사용한다. 그다음 새 프로젝트부터 `python3 scripts/init_project.py`를 실행해 `002`, `003` 순으로 만든다. 특정 번호가 필요하면 `python3 scripts/init_project.py --number 7`처럼 실행해 `007`을 만든다. 기존 번호를 덮어쓰지 않는다.

```text
coupang-detail-page/
├── inputs/
│   └── <project-no>/
│       ├── product-info.md
│       ├── original-images/
│       │   ├── front.jpg
│       │   ├── side.jpg
│       │   └── detail.jpg
│       └── real-references/
│           ├── reference-01.jpg
│           └── reference-02.jpg
└── outputs/
    └── <project-no>/
        ├── raw/
        │   └── retries/
        ├── final/
        ├── fact-ledger.md
        └── prompt-set.md
```

원본 상품 이미지는 반드시 `inputs/<project-no>/original-images/`에 저장한다. 실제 상세페이지 스타일 레퍼런스는 `inputs/<project-no>/real-references/`에 저장한다. 검증된 상품 정보는 같은 프로젝트의 `product-info.md`에 입력한다. 상품 실행용 이미지를 스킬 자체의 `references/`에 넣지 않는다. 그 폴더에는 작업 지침 문서만 둔다.

`inputs/*/product-info.md`, 원본 사진, 스타일 레퍼런스와 생성 결과는 `.gitignore`로 제외되는 실행 데이터다. 실제 상품 정보가 담긴 `product-info.md`를 스킬 저장소에 강제로 추가하거나 커밋하지 않는다. `product-info.example.md`만 배포용 빈 양식이다.

로컬 이미지가 다른 위치에 있으면 원본 파일을 이동하거나 덮어쓰지 말고 표준 입력 폴더로 복사한다. 같은 파일명이 이미 있으면 `-02`, `-03`과 같은 접미사를 붙인다. 대화에만 첨부된 이미지를 로컬 파일로 복사할 수 없으면 해당 실행에서는 대화 이미지를 사용하고, 로컬에 저장했다고 보고하지 않는다.

`raw/` 최상위에는 장별 승인 후보 `page-01.png`부터 `page-10.png`를 둔다. 실패한 생성본과 교체 전 파일은 `raw/retries/`에 보관한다. `final/`에는 규격까지 통과한 최종본 10개만 둔다.

### 2. 모든 입력 이미지를 검사한다

- 실행할 프로젝트 번호를 확정하고 `inputs/<project-no>/product-info.md`를 읽는다.
- `inputs/<project-no>/original-images/`와 `inputs/<project-no>/real-references/`의 파일 목록을 먼저 확인한다.
- `python3 scripts/check_project.py <project-no>`를 실행한다. 로컬로 복사할 수 없는 대화 첨부 원본만 있다면 실제 첨부 개수를 세어 `python3 scripts/check_project.py <project-no> --conversation-originals <count>`로 실행하고, 로컬에 저장되지 않았음을 보고한다. 필수 상품 정보나 어느 형태의 원본 이미지도 없으면 `image_gen`을 호출하지 말고 누락 항목과 저장 위치를 보고한다.
- 로컬 이미지는 `view_image`로 직접 확인한다.
- 각 이미지를 `ORIGINAL_PRODUCT` 또는 `STYLE_REFERENCE` 중 하나로 명시한다.
- 원본별 시점, 보이는 부품, 색상, 재질, 라벨과 구성품을 기록한다.
- 실제 레퍼런스별 팔레트, 여백, 카드 구조, 조명, 카피 리듬만 추출한다.
- 레퍼런스 속 경쟁 상품, 문구, 로고, 가격, 리뷰, 인증, 수치, 배지와 쿠팡 UI는 사용하지 않는다.

### 3. 사실·카피 원장을 고정한다

`fact-ledger.md`에 다음 세 묶음을 작성한다.

- `allowed facts`: 사용할 수 있는 상품 사실, 구성품, 수치와 인증
- `approved copy`: 장별로 그대로 렌더링할 정확한 한글 문구
- `forbidden claims`: 사용자 금지 표현과 자동 금지 표현

텍스트뿐 아니라 시각적 암시도 원장에 근거하게 한다. 예를 들어 방수 정보가 없으면 물속 사용, 물방울 배지, 방수 아이콘도 넣지 않는다.

가격, 할인율, 리뷰, 별점, 판매량, 수상, 보증, 테스트 결과, 의학적 효능을 입력 없이 만들지 않는다. `100%`, `완벽`, `무조건`, `최고`, `1위`, 치료·완치·회복·개선 보장 표현을 근거 없이 사용하지 않는다.

### 4. 하나의 디자인 시스템과 10개 프롬프트를 만든다

생성 전에 다음 공통 블록을 한 번 확정하고 `prompt-set.md`의 모든 장에 동일하게 적용한다.

- 색상 팔레트와 배경
- 제품 촬영 또는 렌더링 방식
- 조명과 그림자 강도
- 카드 모양, 모서리, 아이콘 스타일
- 헤드라인 크기, 본문 밀도와 여백
- CTA 스타일
- 제품 불변 요소와 공통 금지 요소

[references/real-references.md](references/real-references.md)의 10장 청사진에 따라 장별 목적, 정확한 카피, 장면과 구도를 다르게 만든다. 한 장당 헤드라인 1개와 보조 문구 1~3개만 사용한다.

### 5. `image_gen`으로 10장을 순서대로 생성한다

- 각 장을 독립된 `ads-marketing` 이미지로 호출한다.
- 서로 다른 장을 `n` 변형이나 한 장의 콜라주로 생성하지 않는다.
- 모든 호출에 동일한 원본 상품 이미지 세트와 동일한 디자인 시스템 블록을 다시 제공한다.
- 이전 생성본을 제품 사실의 기준으로 사용하지 않는다. 누적된 외형 변형을 막기 위해 항상 원본 상품 이미지를 기준으로 삼는다.
- 로컬 파일로 모든 입력을 전달할 수 있으면 `referenced_image_paths`에 필요한 원본과 스타일 레퍼런스를 함께 넣는다.
- 대화에만 있는 입력을 사용할 때는 필요한 모든 이미지를 포함하는 가장 작은 `num_last_images_to_include` 값을 사용한다. 대화 전용 이미지는 한 호출에 최대 5장만 전달할 수 있으므로 6장 이상이면 핵심 원본을 최대 5장으로 선별하거나 나머지를 로컬 입력 폴더에 복사한 뒤 진행한다.
- `referenced_image_paths`와 `num_last_images_to_include`를 동시에 사용하지 않는다. 어느 방식으로도 모든 입력을 포함할 수 없으면 누락 이미지를 다시 첨부해 달라고 요청한다.
- 각 호출 뒤 선택한 결과를 built-in 기본 저장 위치에만 남기지 말고 `outputs/<project-no>/raw/page-NN.png`로 복사한다. 기존 파일을 교체할 때는 먼저 `outputs/<project-no>/raw/retries/`로 옮겨 재시도 이력을 보존한다.
- `prompt-set.md`에 장 번호, 역할, 최종 프롬프트, 원본 경로, 스타일 레퍼런스 경로, 허용 사실, 출력 경로와 생성 시도 번호를 기록한다.

각 호출 프롬프트를 다음 순서로 구성한다.

```text
사용 사례: ads-marketing
자산 유형: 쿠팡 모바일 상품 상세페이지, 내부 작업 순서 <10장 중 N장>, 이미지 안에는 장 번호를 렌더링하지 않음
핵심 요청: <이 장의 구매 설득 목적>
입력 이미지 역할: ORIGINAL_PRODUCT=<제품 정체성 기준>, STYLE_REFERENCE=<스타일 참고만>
제품 불변 요소: <형태, 비율, 색상, 재질, 부품, 라벨, 구성품>
장면과 배경: <장면과 배경>
스타일과 표현 방식: <고정 디자인 시스템>
구도와 프레이밍: 780x3000을 목표로 하는 초세로형 13:50 캔버스, 모든 핵심 제품과 텍스트를 좁은 중앙 안전 영역 안에 배치하고 바깥쪽 여백에는 확장 가능한 배경만 사용
정확한 문구: "<승인된 한글 카피>"
필수 제약: 따옴표 안의 문구만 렌더링, ORIGINAL_PRODUCT 보존, STYLE_REFERENCE는 추상적인 시각 문법에만 사용, 모바일에서 읽히는 정보 위계
금지 요소: 근거 없는 주장, 추가 문구, 깨진 한글, 워터마크, 페이지 번호, 쿠팡 로고·앱 UI, 검색·장바구니·하트 아이콘, 스마트폰 목업, 레퍼런스의 상품·로고·카피·가격·리뷰·인증·배지·수치 복제
```

### 6. 장별로 즉시 검사하고 필요한 장만 재생성한다

각 결과를 저장한 뒤 `view_image`로 검사한다.

- 상품의 실루엣, 비율, 색상, 재질, 부품 수, 라벨 위치와 구성품이 원본과 일치하는지 확인한다.
- 승인 카피와 이미지 속 한글을 글자 단위로 대조한다. 자모 누락, 띄어쓰기·숫자 변경, 더미 영문, 추가 문구를 실패로 처리한다.
- 해당 장의 목적과 다음 장으로 이어지는 흐름이 분명한지 확인한다.
- 금지된 UI, 로고, 페이지 번호, 허위 정보와 시각적 효능 암시가 없는지 확인한다.

실패하면 원인을 하나만 지정해 해당 장을 다시 호출한다. 텍스트 수정 재시도에서도 실패하면 텍스트 없는 버전과 명확한 빈 카피 영역을 생성한 뒤 `scripts/run_overlay_copy.py`로 승인 카피를 결정론적으로 삽입한다. 실제 후편집을 처음 실행할 때 스킬 로컬 `.runtime/`에 격리된 Pillow 환경이 필요하면 자동 준비된다. 예: `python3 scripts/run_overlay_copy.py text-free.png page-01.png --text "승인된 문구" --box 0.1,0.08,0.8,0.2`. 한글 폰트를 자동으로 찾지 못하면 `--font <한글-font.ttf-or-ttc>`를 지정한다. 빈 영역이나 깨진 문구가 남은 이미지는 최종본으로 승인하지 않는다.

### 7. 규격을 정규화하고 하드 게이트를 실행한다

built-in `image_gen`은 정확한 픽셀 크기 인자를 제공하지 않으므로 프롬프트에 `780x3000`을 적은 것만으로 규격 충족을 선언하지 않는다.

- 원본 비율이 `13:50`과 실질적으로 같으면 균일 리샘플링으로 `780x3000`으로 변환한다.
- 비율이 다르면 비균일 스트레칭을 금지한다. 먼저 결과를 시각적으로 검사하고 모든 핵심 요소가 중앙 안전 영역 안에 있을 때만 중앙 크롭한다. 그렇지 않으면 해당 장을 재생성한다.
- macOS에서는 `python3 scripts/normalize_pages.py outputs/<project-no>/raw outputs/<project-no>/final`을 사용한다. 안전한 중앙 크롭을 직접 확인한 경우에만 `--allow-center-crop`을 추가한다.
- `python3 scripts/validate_pages.py outputs/<project-no>/final`이 성공할 때까지 완료하지 않는다.

## 최종 검수

다음을 모두 만족해야 완료한다.

- 최종 폴더에 이미지가 정확히 10개 있고 `page-01`부터 `page-10`까지 일대일 대응한다.
- 10개 모두 실제 크기가 `780x3000`이다.
- 같은 파일이 중복 저장되지 않았고 각 장의 장면과 역할이 구분된다.
- 10장이 후킹, 필요성, 상품 소개, 상황 비교, 장점 1·2, 장점 3, 차별점 비교, 추천 대상, 활용 장면, CTA 순서에 대응한다.
- 제품이 보이는 모든 장에서 원본 외형과 구성품이 일관된다.
- 실제 레퍼런스의 상품, 브랜드, 카피와 사실 정보가 복제되지 않는다.
- 승인 카피가 정확하고 모바일에서 읽을 수 있다.
- 입력에 없는 가격, 할인, 리뷰, 별점, 판매량, 수치, 인증, 효능과 보증이 없다.
- 쿠팡 로고, 앱 UI, 검색·장바구니·하트 아이콘, 스마트폰 목업과 이미지 내부 페이지 번호가 없다.

마지막 응답에 프로젝트 번호, 최종 이미지 10개의 저장 경로, `prompt-set.md` 경로, 사용한 입력 이미지 역할, 재생성한 장, 검증 결과를 간결하게 보고한다.
